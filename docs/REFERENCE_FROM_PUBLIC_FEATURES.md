# Reference: Public devcontainer-features and Best Practices

This document summarizes findings from official and community devcontainer-features repositories and how they can inform this repo.

## Sources

- **devcontainers/features** (official): https://github.com/devcontainers/features  
- **devcontainers/feature-starter** (template): https://github.com/devcontainers/feature-starter  
- **Best practices**: https://devcontainers.github.io/guide/feature-authoring-best-practices

## 1. Official Node feature (devcontainers/features)

- **Path**: `src/node/`
- **Approach**: Uses **NVM** (Node Version Manager), not NodeSource or distro packages.
- **Key details**:
  - Installs NVM to `/usr/local/share/nvm` (configurable).
  - Sets `containerEnv` in `devcontainer-feature.json` so the container has node/npm on PATH:
    ```json
    "containerEnv": {
      "NVM_DIR": "/usr/local/share/nvm",
      "NVM_SYMLINK_CURRENT": "true",
      "PATH": "/usr/local/share/nvm/current/bin:${PATH}"
    }
    ```
  - Supports Debian and RHEL-family; uses `/etc/os-release` for detection.
  - Uses `check_packages()` and `pkg_mgr_update()` for idempotent package installs.
  - Yarn: Corepack first, then npm global fallback.
  - Non-root user: uses `USERNAME` / `_REMOTE_USER` and `su ${USERNAME} -c "..."` for nvm install.

**Takeaway for this repo**: Our features depend on Node and either use the Node feature (scenarios) or install Node themselves (autogenerated tests). We do not use NVM; we use NodeSource + distro + corepack. That’s a valid alternative; the important part is documenting it and having a clear fallback (which we do).

## 2. Features that depend on Node

- **installsAfter**: Declaring `"installsAfter": ["ghcr.io/devcontainers/features/node"]` is the standard way to express “run after Node.”  
- **Publish/test behavior**: When the devcontainer CLI runs feature tests or publish, it may run each feature in isolation (no other features). So our install scripts must be able to install Node when it’s missing (which we do with NodeSource + distro + corepack).
- **copilot-cli** (official): Does not use Node; downloads a binary. So it’s not a direct pattern for “npm install -g” CLIs.

**Takeaway**: Our pattern (installsAfter node + optional self-install of Node when missing) is consistent with how dependency features are used and tested.

## 3. Best practices (from official guide)

| Practice | Recommendation | This repo |
|----------|----------------|-----------|
| **Testing** | Use `devcontainer features test` with multiple base images and scenarios. | We use test.yaml with matrix (features × baseImage) and scenarios. |
| **Idempotency** | Feature should tolerate being installed multiple times / with different options. | Our installs are effectively idempotent (npm install -g is re-runnable). |
| **PATH** | Use `containerEnv.PATH` if the feature installs to a custom directory. | We install to system paths (/usr/bin) or npm global; no custom PATH needed. |
| **OS / platform** | Detect OS and fail with a clear message on unsupported distros. | We detect package manager (apt/apk/dnf/yum) but don’t explicitly fail on “unknown” with a message; we could add that. |
| **Redundant strategies** | Prefer multiple install paths (e.g. upstream then fallback) to improve resilience. | We already do: NodeSource → distro nodejs/npm → corepack + npm wrapper. |
| **Root** | Scripts run as root; consider `_REMOTE_USER` / `_CONTAINER_USER` if writing into user dirs. | We install globally; no user-dir logic. |

## 4. Official test setup (devcontainers/features)

- **Node scenarios**: Many scenarios for different distros (debian 11/12/13, ubuntu, almalinux, rocky, fedora, mariner) and options (version, pnpm, additional versions).
- **Test workflow**: Matrix of features × base images (e.g. ubuntu bionic/focal/jammy, debian 11/12, devcontainers base).
- **Scenarios vs autogenerated**: Scenarios define explicit `image` + `features`; autogenerated tests use a single feature per run with a base image.

**Suggestions for this repo**:
- Keep current matrix; optionally add more base images (e.g. `debian:12`, `ubuntu:22.04`) for reproducibility.
- Scenarios already use `ghcr.io/devcontainers/features/node:1` + our feature; we could add one scenario without the node feature to exercise our “install Node when missing” path.

## 5. feature-starter (install.sh pattern)

- Uses `#!/bin/sh` and `set -e`.
- Root check: `if [ "$(id -u)" -ne 0 ]; then ... exit 1; fi`.
- Writes to `/usr/local/bin` and `chmod +x`.
- Uses env vars for options (e.g. `GREETING=${GREETING:-undefined}`).

Our install scripts already use `#!/bin/sh`, `set -e`, and option-like behavior; we could add an explicit root check and document env vars in README/NOTES.

## 6. Concrete improvement ideas

1. **devcontainer-feature.json**
   - Ensure all features have `documentationURL` and, if applicable, `licenseURL` (we already do).
   - Optional: add `options` for CLI version (e.g. `version` with default `"latest"`) for consistency with other features.

2. **install.sh**
   - Add an explicit root check at the top (as in feature-starter and official features).
   - On unsupported package manager, exit with a clear message (e.g. “Unsupported package manager: $PKG_MANAGER”) instead of only “WARNING”.

3. **Tests**
   - Optionally add a scenario that does *not* include the Node feature (e.g. `debian:latest` + only our feature) to regularly test the “install Node when missing” path.
   - Optionally add `debian:12` or `ubuntu:22.04` to the base image matrix for more stable versioning.

4. **Docs**
   - In README/NOTES, state that the feature can be used with or without the Node feature; when used without, Node (and npm/corepack) are installed automatically when possible.

---

## 7. Summary

- **Official Node**: NVM-based, `containerEnv` for PATH; we use NodeSource + distro fallback (no corepack/shim) and require the Node feature when auto-install is not available.
- **installsAfter** node is the recommended usage; scenarios always include the Node feature.
- **Simplified design**: Root check kept. No `run_npm`, no corepack, no shim wrapper. Node = NodeSource try then distro `nodejs`/`npm` and `node` symlink only. Single scenario per feature (with Node).
